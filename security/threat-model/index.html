    Threat Model¶

    This document outlines the security assumptions, adversarial capabilities, and trust model of the ZkSnap protocol.

    System Model¶

    Protocol Participants¶

    1. Voters
    2. Hold registration and voting keys
    3. Create and submit votes

    4. Verify results

    5. Coordinator

    6. Processes votes
    7. Maintains state

    8. Generates proofs

    9. DRand Service

    10. Provides timelock encryption
    11. Distributed trust
    12. Time-based decryption

    Trust Assumptions¶

    Coordinator Trust Model¶

    1. Semi-Honest Model
    2. Follows protocol correctly
    3. May attempt to learn vote contents
    4. Cannot forge votes or proofs

    5. Cannot prevent valid vote inclusion

    6. Capabilities and Limitations 

      Can:
- View encrypted votes
- Process submissions
- Generate proofs
- Manage state

Cannot:
- Decrypt votes early
- Forge valid votes
- Link votes to voters
- Prevent valid voting

    DRand Assumptions¶

    1. Security Model 

      Assumes:
- Threshold security
- Byzantine fault tolerance
- Distributed trust
- Correct time maintenance

    2. Trust Requirements 

      - Majority honest participants
- Correct time-lock enforcement
- Proper key revelation
- Protocol adherence

    Adversarial Model¶

    Adversary Capabilities¶

    1. Network Control 

      Can:
- Observe all communications
- Delay messages
- Read public data
- Track timing

Cannot:
- Modify messages
- Block communications entirely
- Break encryption

    2. Computation Power 

      Assumed Limits:
- Polynomial-time computation
- No quantum capabilities
- Standard cryptographic bounds
- Cannot break timelock

    Attack Vectors¶

    1. Vote Privacy Attacks 

      Types:
- Vote content analysis
- Timing analysis
- Correlation attacks
- Identity linking

    2. Protocol Attacks 

      Categories:
- Double voting attempts
- Invalid vote insertion
- State corruption
- Proof forgery

    Security Properties¶

    Vote Privacy¶

    1. Content Privacy 

      Guarantees:
- Vote secrecy until tally
- No partial information leakage
- No statistical inference

    2. Voter Privacy 

      Ensures:
- Voter anonymity
- Action unlinkability
- Participation privacy

    Vote Integrity¶

    1. Correctness 

      Properties:
- One vote per voter
- No vote modification
- Accurate tallying

    2. Verifiability 

      Features:
- Public verification
- Proof validation
- Result auditability

    Attack Scenarios¶

    Network-Level Attacks¶

    1. Communication Attacks 

      Types:
- Message interception
- Timing analysis
- Replay attacks
- Man-in-the-middle

    2. Mitigation 

      Methods:
- Encryption
- Authentication
- Replay protection
- Timing resistance

    Protocol-Level Attacks¶

    1. Vote Manipulation 

      Attempts:
- Invalid vote insertion
- Vote modification
- Double voting
- State corruption

    2. Countermeasures 

      Protections:
- ZK proofs
- Nullifier system
- State verification
- Proof validation

- Forward security

    2. Integrity Properties 

      - Vote correctness

- Proof soundness
